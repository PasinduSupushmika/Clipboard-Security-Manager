1. Introduction 
1.1 Purpose  
This document specifies the software requirements for Clipboard Security Manager (CMS) v1.0, a 
cross-platform background application designed to encrypt and control clipboard data. The CSM intercepts 
all copy/paste operations. On “Copy” it immediately encrypts the clipboard contents, and on “Paste” it 
prompts for user authentication before decrypting the data. The goal is to prevent any unauthorized app or 
user from accessing sensitive clipboard information. This scope includes the entire CSM system (Encryption 
engine, Authentication module, user dialogs and loggings) but doesn’t cover changes to the underlying OS 
itself. In current mobile an desktop Oses, the clipboard typically has Weak or no access controls. Once the 
text is copied, any app in the foreground can read it. CSM’s design secures this shared resource by applying 
strong cryptography on the copied data (AES-256) and requiring a password check (Argon2id) on paste, 
thereby turning the clipboard into a protected vault rather than a public buffer.

1.3 Project Scope 
Clipboard Security Manager is the software product being specified. Its purpose is to protect 
sensitive clipboard data in accordance with corporate security goals. By encrypting any copied data and 
gating paste operations behind strong authentication, CSM prevents the data leakage that researchers have 
identified as a major privacy risk. For example, studies show that mobile Oss allow all foreground apps to 
access clipboard content without permission [1],[2], enabling data-stealing attacks. CSM’s objectives are 
therefore to close this gap. Key benefits include: 
• Prevent Data Leakage: Encrypt clipboard contents on copy so that other apps (or Malware) cannot

• Enforce Authentication: Require user password entry on paste (verified with Argon2id hashing [3]) 
to ensure only authorized use.

intellectual property that might otherwise be exfiltrated via copy/paste. [1] [2] Software Requirements Specification for Clipboard Security Manager Page 2 
 
• Use Standard Cryptography: Leverage industry – standards algorithms (AES – 256 is a NIST – 
approved block cipher [4], and Argon2id is the 2015 PHC winner designed to resist cracking [3]) to 
meet security requirements. 
In summary, CSM v1.0 addresses the company’s strategic goal of safeguarding data in user workflows 
(especially in context like copying passwords or confidential text). Its scope covers the full clipboard –

2. Overall Description 
2.1 Product Perspective 
Clipboard Security Manager (CSM) is a new, standalone security application rather than a follow-on 
to an existing product. It runs as a client-side background service on user devices (Windows, Linux, macOS) 
to enforce encryption and access control on the system clipboard. This product fills a gap left by all major 
operating systems which is the clipboard is inherently globally accessible and insecure, with minimal built-in 
access control. [1] [2] Unlike existing enterprise DLP or OS components, CSM is designed specifically to sit 
alongside the OS and intercept copy/paste events for any application. 
 CSM is not part of a larger software suite. Instead, it integrates with standard OS clipboard APIs to 
monitor clipboard usage. Conceptually, CSM can be viewed as an “add-on” security layer. It hooks the 
clipboard subsystem, encrypts data on copy, and unlocks it on paste only after authentication. The functional 
interfaces include the OS clipboard service (for all copy/paste requests) and the local user interface (for 
password prompts). In system architecture, CSM operates independently, the OS and other applications need 
not be modified. For example, research shows clipboard vulnerabilities hash shown that attackers can 
manipulate or steal shared clipboard data in both local and remote sessions [1][2][3], highlighting the need 
for a client-side protector like CSM. 
 A simple block diagram would show CSM as a layer between the user’s applications and the OS 
clipboard. Applications issue copy/paste as usual; CSM transparently encrypts data being placed on the 
clipboard, and when the user pastes, CSM prompts for a password before decrypting and forwarding the data. 
No central server or cloud component is required. All encryption and decryption is done locally. In summary, 
CSM is a self-contained security middleware that augments the existing OS clipboard with strong 
confidentiality and authentication controls.

Security/Environmental Constraints:  
CSM runs with normal user privileges. It does not require administrator rights except possibly for

monitor keystrokes. Copy/paste under remote desktop or virtual machines propagates through

2.4 Design and Implementation Constraints 
The design of Clipboard Security Manager (CSM) is constrained first by the nature of clipboard 
systems themselves. Prior research shows that clipboard data can be exposed to foreground applications, 
monitored by malicious processes, and abused across remote desktop or in-memory attack paths [1] [2] [5]. 
Because of this, CSM must operate as a user-space protective layer around the clipboard rather than assume 
privileged access to the operating system’s internal clipboard mechanisms. The software must intercept copy 
and paste actions, secure copied content immediately, and enforce user verification without altering the 
clipboard architecture itself [1] [5].  
A second major constraint is cross-platform implementation. The project is intended to run on 
Windows, Linux, and macOS, but clipboard behavior differs across these operating systems in event 
handling, background execution, and session isolation. The implementation therefore needs platform-specific 
adapters or abstraction layers so that the same security workflow can be preserved across all supported 
environments [1] [5]. Clipboard exposure in remote desktop and virtualized environments further increases 
complexity, because clipboard data may move through host guest or session-based channels beyond the local 
desktop context [5].  
A third constraint is cryptographic correctness and parameter selection. CSM depends on AES-256 
for encryption and Argon2 for password hashing, but secure deployment requires correct use of these 
primitives. NIST’s review of AES confirms its continued role as a standard block cipher, while the Argon2 
literature shows that implementation quality and parameter selection strongly affect real-world effectiveness 
[4] [3]. The system must therefore avoid weak key handling, minimize plaintext exposure, and ensure that 
encryption and verification are implemented safely and consistently [3] [4].  
A further constraint is performance overhead. Clipboard protection occurs during frequent user 
actions, so the application must remain responsive when encrypting copied data, prompting for 
authentication, logging events, and handling alerts. Studies of clipboard usage show that clipboard access is 
frequent in ordinary workflows, which makes latency and efficiency important design requirements rather 
than optional optimizations [1] [5]. The software should therefore use efficient background processing and 
avoid unnecessary polling, large memory usage, or delayed user feedback.  
The software is also constrained by usability and security balance. Since the proposed workflow 
introduces confirmation prompts and password checks before pasting, the design must prevent abuse without 
making normal copy-paste tasks unusable. Research on clipboard access shows that blanket blocking is not 
always sufficient and that user-visible warnings or notifications may be necessary, but these controls still 
need to remain understandable and minimally disruptive [1]. The CSM workflow should therefore trigger 
security prompts only when required and keep the escalation logic clear to the user [1] [5].  
There are also constraints related to trust boundaries and data handling. Clipboard attacks may 
originate locally or through remote-session channels, which means the system must assume the surrounding 
environment is not fully trusted [5]. As a result, sensitive values such as passwords, encrypted clipboard 
payloads, and audit records must not be exposed in logs, temporary files, or unnecessary memory buffers. 
This is especially important because a secure algorithm alone does not guarantee security if the 
implementation leaks data through poor configuration or handling [3] [4] [5].  Software Requirements Specification for Clipboard Security Manager Page 6 
 
Finally, the implementation is constrained by the scope of the project itself. CSM is a defensive 
clipboard-protection subsystem, not a full enterprise data-loss-prevention platform. Its implementation must 
therefore stay focused on clipboard encryption, authentication, alerting, and logging, while broader 
organizational controls remain outside the system boundary. This keeps the project feasible while still 
addressing the clipboard security gap demonstrated in the literature [1] [2] [5].  
 
2.5 Assumptions and Dependencies 
The CSM assumes that the target desktop operating systems continue to expose clipboard and 
session-event interfaces that can be observed from a normal user-space background process. This is 
important because clipboard data can be shared across local apps as well as remote desktop sessions, and 
remote clipboard transfer is explicitly mediated through clipboard virtual-channel behavior in RDP 
environments [1] [5] [8]. If a platform restricts clipboard interception or session monitoring more 
aggressively, some protection features may be reduced or unavailable. 
The system also assumes the software will run on modern 64-bit desktop hardware with enough CPU 
and memory to perform frequent encryption and password-verification tasks without noticeable delay. This is 
a reasonable assumption because AES remains a standard-reviewed block cipher and modern 64-bit 
processors commonly support AES acceleration, while Argon2id is the mandatory support variant in RFC 
9106 and is designed specifically for password hashing and memory-hard protection [4] [6] [7]. 
A further assumption is that secrets are not stored in plain text and that the deployment environment 
can support a secure local secret-handling method. This is necessary because classical stored keys are a 
known attack target, and the cryptographic literature treats permanently stored digital keys as a major 
weakness unless they are protected or replaced by stronger storage approaches [9] [10]. For CSM, this means 
any key material, salts, or sensitive configuration data should be protected through OS-backed or hardware-
backed storage rather than ordinary files. 
The project additionally assumes that optional notification features such as email alerts can rely on a valid 
outbound network path and a functioning message-delivery service. If that dependency is unavailable, the 
core clipboard-protection workflow should still operate locally, but alerting and remote notification features 
may need to be disabled or deferred. 
In terms of dependencies, CSM depends on the operating system clipboard subsystem, cryptographic 
libraries implementing AES and Argon2id correctly, and a secure local storage mechanism for configuration 
and secret material [6] [7] [8] [9]. If the alerting module is included in the final build, it also depends on a 
mail or notification service. If the project later reuses components from another implementation, those reused 
modules must remain compatible with the clipboard security workflow and must not introduce plaintext 
storage or weak key handling [9] [10].

4. External Interface Requirements 
4.1 User Interfaces 
The software product should provide a secure, intuitive, and user-friendly graphical user interface 
(GUI) that enables users to interact efficiently with the system while maintaining strong security and privacy 
protections. Given the system’s focus on clipboard monitoring and data protection, the interface should be 
designed to clearly communicate security-related events and alerts to users without causing confusion or alert 
fatigue. 
The user interface shall support multiple user roles such as administrators and general users. Each 
role shall have controlled access to system functionalities based on the principle of least privilege, ensuring 
that users can only access features relevant to their responsibilities. Sensitive operations shall require 
additional confirmation or authentication steps. 
The interface shall follow consistent design standards across all screens, including uniform color 
schemes, typography, icons, and layout structures. Security-related visual indicators (such as warnings, 
alerts, and encryption status) shall be clearly distinguishable using standardized colors (e.g., red for threats, 
green for secure states, and yellow for warnings). This consistency improves usability and reduces the risk of 
user error [11]. 
The system shall include the following key interface components: 
• Login and Authentication Interface 
Provides secure access using credentials protected with strong hashing mechanisms such as Argon2 
[6]. Multi-factor authentication may be integrated for enhanced security.  
• Dashboard Interface 
Displays real-time system status, clipboard activity summaries, detected threats, and security alerts. 
Users shall be able to quickly assess whether their data is secure.  
• Clipboard Monitoring Interface 
Shows current and historical clipboard data activity, including detected access attempts by 
applications. Sensitive data exposure shall be highlighted based on risk levels, aligning with known 
clipboard attack vectors [2] [5].  
• Alerts and Notification Interface 
Provides real-time notifications for suspicious activities such as unauthorized clipboard access or 
data leakage attempts. Alerts shall be clear, actionable, and prioritized based on severity.  
• Data Management Interface 
Allows users to view, filter, and manage stored logs of clipboard activities. Search and filtering 
capabilities shall be provided for efficient analysis.  
• Settings and Security Configuration Interface 
Enables users to configure encryption options (e.g., AES-based protection [4]), clipboard monitoring 
sensitivity, access controls, and notification preferences.  
• Reports and Analytics Interface 
Presents visual summaries (charts, logs, timelines) of clipboard usage patterns and detected threats to 
support analysis and decision-making.  
All input elements (text fields, dropdowns, checkboxes, etc.) shall include validation mechanisms to prevent 
incorrect or malicious input. Error messages shall be clear, concise, and non-technical to ensure usability for 
non-expert users. 
The interface shall follow secure design principles to prevent UI-based attacks such as spoofing or 
clickjacking. Sensitive information displayed on the interface shall be minimized or masked where 
appropriate, in accordance with secure system design practices [9]. Software Requirements Specification for Clipboard Security Manager Page 11 
 
The system shall also ensure responsiveness and adaptability across different devices and screen 
sizes, maintaining usability in both desktop and mobile environments. Important actions such as delete, reset,

Keyboard shortcuts may be provided for advanced users to improve efficiency, particularly for

4.3 Software Interfaces 
The Clipboard Security Manager shall interface with multiple software components to enable 
clipboard monitoring, encryption, authentication, logging, and alerting functionalities. These interfaces are 
designed based on the system’s “Capture – Encrypt – Verify – Paste” architecture. 
 
4.3.1 Operating System Interfaces: 
The system shall interact with the host operating systems (Windows, Linux, macOS) through native 
clipboard APIs and system hooks to monitor and control clipboard activity. 
• On Windows, the system shall use Win32 Clipboard APIs (Ex: OpenClipboard, GetClipboardData, 
SetClipboardData)  
• On Linux, clipboard interaction shall be handled through X11/Wayland services  
• On macOS, the system shall use NSPasteboard APIs  
These interfaces enable: 
• Detection of copy and paste events  
• Retrieval of clipboard data  
• Replacement of clipboard content with encrypted data  
Clipboard interaction is event-driven and occurs in real time, which is critical due to known clipboard 
vulnerabilities [2] [5].

Data flow: 
• Input: Plain-text clipboard data

5. Other Nonfunctional Requirements 
5.1 Performance Requirements 
The Clipboard Security Manager is designed as a lightweight, real-time background application that 
ensures strong security without compromising system performance or user experience. The system shall 
perform encryption of clipboard data immediately after a copy event is detected, with a maximum acceptable 
latency of approximately 100 milliseconds under normal operating conditions. This requirement ensures that 
the encryption process is effectively invisible to the user while still securing sensitive data in memory. 
Similarly, the decryption and authentication process during paste operations, including password verification 
using Argon2, shall be completed within 200 milliseconds to maintain a seamless interaction flow while 
preserving resistance to brute-force attacks [6]. 
To ensure efficiency, the system shall maintain low resource consumption, with CPU utilization 
remaining within 5–10% during active monitoring and memory usage kept below 150 MB under typical 
workloads. Clipboard monitoring shall operate continuously in real time, ensuring that no clipboard events 
are missed. This is particularly important given the rapid exploitation window associated with clipboard-
based attacks, where malicious processes can access data immediately after it is copied [2] [5]. The use of 
AES-256-GCM encryption further supports performance requirements by providing a well-established 
balance between computational efficiency and cryptographic strength [4]. 
Additionally, logging and data storage mechanisms shall be optimized to minimize disk input/output 
operations, especially considering the implementation of a 30 day rolling log policy. The system shall also 
support multiple consecutive clipboard operations without performance degradation, ensuring stability even 
under frequent user interaction. 
 
5.2 Safety Requirements 
The Clipboard Security Manager shall ensure that its operation does not result in unintended data 
loss, system instability, or disruption to normal user workflows. Since the application actively modifies 
clipboard behavior, it is critical that clipboard data always remains accessible and recoverable. In the event 
of a system failure, crash, or unexpected termination, the system shall fail in a safe manner, ensuring that 
clipboard data does not remain permanently encrypted or inaccessible to the user. 
The application shall preserve the integrity of clipboard contents throughout the encryption and 
decryption processes, avoiding any corruption or unintended modification of user data. Furthermore, the 
system shall not interfere with essential operating system functions or other running applications. All 
security-related interactions, such as authentication prompts and confirmation dialogs, shall be designed to 
prevent accidental user actions that could lead to data exposure, while still allowing users to maintain control 
over their workflow. 
Error handling mechanisms shall be implemented to ensure that any faults are managed gracefully, 
with clear and understandable feedback provided to the user. Additionally, the system shall implement 
controlled responses to authentication failures, such as warnings and temporary lockouts, rather than abrupt Software Requirements Specification for Clipboard Security Manager Page 17 
 
termination or data deletion. These design considerations align with established usability and safety 
principles, ensuring that the system minimizes human error and operates reliably under all conditions [11]. 
 
5.3 Security Requirements 
Security is the core objective of the Clipboard Security Manager, and the system shall enforce 
comprehensive measures to protect sensitive data from unauthorized access, leakage, and manipulation. All 
clipboard data shall be encrypted immediately upon capture using AES-256-GCM, ensuring that data stored 
in system memory is never exposed in plaintext form. This approach addresses the inherent vulnerability of 
clipboard architectures, where data is typically accessible to any application with basic permissions [4]. As a 
result, even if malicious software attempts to access the clipboard, it will retrieve only encrypted ciphertext. 
User authentication shall be required before any encrypted data can be decrypted and pasted. 
Passwords shall be securely processed using Argon2 hashing, which provides strong resistance against brute-
force and side-channel attacks due to its memory-hard design [6]. The system shall implement a multi-level 
security escalation protocol to further enhance protection. A first failed authentication attempt shall result in 
a warning message, a second failed attempt shall trigger an automated email alert, and a third failed attempt 
shall result in a temporary account lockout. This layered approach ensures both prevention and detection of 
unauthorized access attempts. 
All external communications, including email notifications and API interactions, shall be secured 
using protocols such as HTTPS and SMTP over TLS to prevent interception and tampering. Sensitive data, 
including clipboard content and authentication credentials, shall never be transmitted over these channels. 
Instead, only minimal metadata, such as timestamps and event descriptions, shall be communicated. 
The system shall also be designed to mitigate known clipboard-related threats, including clipboard 
hijacking, unauthorized monitoring by background applications, and remote data leakage through protocols 
such as Remote Desktop Protocol (RDP) [2] [5] [8]. Furthermore, the system shall address the growing risk 
of unintentional data exposure through web-based platforms, such as AI applications, by enforcing 
authentication before paste operations [1]. 
To support accountability and forensic analysis, the system shall maintain encrypted logs of all 
security-relevant events, including authentication attempts and clipboard access activities. These logs shall 
follow a 30-day rolling retention policy and shall be protected against unauthorized access. The overall 
system design shall adhere to fundamental cybersecurity principles, including confidentiality, integrity, and 
availability, while also following secure-by-design practices and minimizing reliance on exposed secrets [9] 
[10].
